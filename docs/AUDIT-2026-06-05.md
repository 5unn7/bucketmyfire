## Executive summary

This is exceptional solo engineering wrapped around a business that is one push and one button away from being real. The codebase has improved materially since the 6.5/10 audit: the retention keystone (Daily Burn), the cold-load splash, the instant-fly Quick Fly path, real ESLint/Prettier, privacy/terms/manifest, deploy hardening, and an image share-card are all built. The architectural invariants the docs claim still genuinely hold in code — the `sim/` engine-agnostic boundary is airtight across all 9 modules, `World.ts` is a true single-source-of-truth keystone, there are zero `as any` casts, zero TODO/FIXME/HACK markers, and the security posture (anon-key-only bundle, RLS-locked tables, SECURITY DEFINER RPCs, client-side email hashing, XSS-closed-by-construction) is more disciplined than most funded startups ship.

The gap is no longer "is the code good" — it is "do players actually have the good code, and can it make money." Two facts dominate this audit:

1. **Prod is the bad build.** 17 launch-readiness commits are unpushed, and `main` auto-deploys to GitHub Pages. The live site at bucketmyfire.com is therefore missing the privacy policy (a hard legal prerequisite for the stated ad/merch model), Daily Burn, the share-card, Quick Fly, AND the deploy-hardening that would protect the next push. The team built the launch and is sitting on it.
2. **The revenue surface is vapor.** The legal docs already describe a print-on-demand merch store, but no store/merch/Fourthwall CTA exists anywhere in the game — including the win screen the business plan calls the single highest-leverage commercial move. The retention loop, likewise, is a button with no comeback mechanic (no streak, no return reward, no cross-day memory).

On top of those, a large crash/explosion gameplay feature plus a freeform-lake feature sit uncommitted on the working tree, unbacked-up and with zero automated coverage (the campaign verifier's "perfect player" never crashes). The engine got the love; the business and operational surfaces did not.

**The verdict:** the game engine is ~8.5/10. The shippable-business is held back by a stale prod, an unbuilt revenue funnel, and a retention loop that does not yet remember the player. None of the surviving findings are code-integrity bugs — the gates pass against the real tree — they are ship-process and product-completion gaps. Fix the push, wire one button, and add a streak, and this jumps to an 8.

## Codebase Architecture & Integrity — 8.5

- Single-config discipline holds; even the uncommitted WIP adds only config imports, no new Scene/DOM leakage.

## Performance vs the mobile-60fps invariants — 7.0

**Verdict.** The runtime hot path is genuinely well-engineered against the project's own invariants. No shader recompiles after load: HeroFireLights is a fixed pool toggled by intensity, fire/structure mesh pools are built once and only flip `.visible`, the VFX layers are ring-buffered Float32Array Points with conditional GPU uploads, FireFieldTexture is a single in-place DataTexture memcpy. Trees are InstancedMesh with chunk culling + LOD. ChaseCamera/BucketSim/HelicopterSim/Composer use field-level scratch with zero per-frame allocation. QualityTier is a proper adaptive-DPR watchdog with hysteresis. Held below 9 by per-frame HUD allocations, the SMOKE pool overrun, and the cold-start asset weight.

**Strengths.**
- Fixed light pool done correctly (HeroFireLights.ts:24-30) — adds 5 lights once, only sets intensity/position per frame.
- All four particle systems are textbook fixed-size ring buffers gating GPU upload behind `if (anyAlive || visible)` (WaterSpray.ts:151, SmokePlume.ts:259, Embers.ts:157).
- Forest is instanced + chunk-culled with a near/mid LOD swap and module-level scratch (trees.ts:250-263,81-83); density bump tier-gated to med/high (Game.ts:459).
- QualityTier scales ONLY DPR within [floor..cap] with up/down hysteresis and a dead zone; composer skipped entirely on low tier.
- TitleScreen disposes its renderer cleanly and the title→game transition is a page reload, so two WebGL contexts never coexist.

## Security — Client Trust Boundary & Data (OWASP / CSO lens) — 8.0

**Verdict.** A well-reasoned posture, above the bar for the genre. XSS is effectively closed by construction: every DOM helper in ui/theme.ts writes user/server strings via textContent and the Leaderboard renders all server names via textContent; the only three innerHTML sites are static dev-authored SVG/markup. The callsign sanitizer NFC-normalizes and strips control/zero-width/bidi code points. The Supabase data layer is correctly RLS-gated; only the anon JWT ships; email is SHA-256 hashed client-side before leaving the device; the privacy policy matches the real data flow; importProgress validates the untrusted cloud blob. Deductions are inherent to a client-side game: the global board is trivially forgeable and unlocks are client-authoritative — both documented as accepted.

**Strengths.**
- XSS closed by construction (theme.ts:125 textContent; Leaderboard.ts:291,330,337,389,425,593 textContent for server names).
- Callsign sanitizer defeats homoglyph/RTL spoofing by numeric code point (callsign.ts isInvisible 78-91).
- RLS real and correct (scores SELECT+INSERT-only with CHECK bounds, no update/delete; cloud_saves/client_errors locked).
- Email PII SHA-256 hashed client-side before any network call (cloudSave.ts:85-105); privacy.html accurate.

## Top Risks

1. **Prod is stale and legally exposed.** The live site auto-deploys from `origin/main`, which is 17 commits behind local and lacks the privacy policy the game's own monetization model requires, plus the entire Daily Burn / share-card / Quick Fly launch surface (VISION-2, critical).
2. **Unbacked-up working tree.** 17 unpushed commits + a 565-insertion, 9-file uncommitted WIP (crash feature, freeform lakes) + an untracked map editor exist on one disk; a loss or botched reset wipes months of work (CQ-3, ARCH-4).
3. **The revenue funnel is unbuilt.** No merch/store CTA exists in the game despite legal docs describing one and the business plan naming it the single highest-leverage move; the funnel cannot even be measured (GTM-1, VISION-1).
4. **Retention does not remember the player.** Daily Burn has no streak, return reward, or cross-day memory, so a one-session player has no compounding reason to return tomorrow (VISION-3, high).
5. **An untested, uncommitted crash/explosion feature changes the lose condition** with zero automated coverage (the perfect-player verifier never crashes) and no dogfood evidence in the tree (CQ-2).
6. **Launch-blind on weak devices** — the crash beacon is built but unconfigured in prod, and one uncaught AudioContext throw can blank-screen the game on long-tail WebViews (VISION-4, CQ-1).
