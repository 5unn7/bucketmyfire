## Problem Statement

The home-screen live wildfire tracker (`src/three/livefire/` + `src/three/ui/home/menus.ts`)
has accumulated bandaid workarounds. The owner's words: *"Cannot see factual infos, missing
many fires, active fire shows nothing, fire weather overlay opacity to be reduce so we can see
the map under… we need to treat this as actual real guide, trust is utmost, accuracy is utmost.
Need to know when data is last updated. Total number is what of this year, all years etc."*

Today the tracker fetches four independent keyless feeds (CWFIS hotspots, CIFFC active fires,
CWFIS burn perimeters, CWFIS FWI raster) plus a CIFFC summary strip. Every feed degrades
**silently to blank** on any failure, so "feed is down," "no fires," and "this layer is empty"
all render identically — as nothing. Freshness is shown as *time since we fetched*, not *time
since the source published*. That combination is precisely what makes a real-data screen feel
broken and untrustworthy.

**Decided framing (this session):** This is an **Honest Window into real fire** — an ambient,
accurate, honestly-labeled view of real wildfire activity that grounds the game in reality. Trust
means: never show fake / wrong / silently-stale data; always say where each layer came from and
how old it really is. It is **explicitly not** an emergency tool — it carries a visible "not for
emergency decisions, see official sources" line and links out to authoritative origins. Scope
stays game-sized; we do **not** build a CIFFC / firesmoke.ca clone.

## What Makes This Cool

- **Honesty as the hero, not the fine print.** A "source ledger" panel shows every feed, its live/
  cached/down status, and exactly how old its data is, each linked to the authoritative origin. A
  game-home fire map that is *visibly more honest than a generic dashboard* is a genuinely novel,
  on-brand thing. It turns "trust is utmost" from a goal into a feature you can point at.
- **It's real, and it's home turf.** The game is northern Saskatchewan; the map shows fires actually
  burning in Saskatchewan and Canada right now. That ties the fiction ("Fight the fire") to the real
  world without a single fabricated pixel.
- **The smoke layer breathes.** A firesmoke.ca-style animated surface-smoke forecast (clearly labeled
  a *forecast*) gives the screen motion and a "whoa" without faking anything.

## Constraints

- **No-scroll single-viewport law (hard rule).** Stat strip + layer control + legend + map + detail
  sheet + a smoke time-scrubber must all fit a phone with **zero page scroll**. Layers move behind a
  compact control (a layer sheet / grouped toggles), not an ever-growing chip row.
- **Solo maintenance.** Every source added is a feed that can break and that one person has to keep
  alive. Favor structured, stable, keyless, CORS-`*` endpoints; treat fragile sources as opt-in and
  loudly honest when stale.
- **Keyless + CORS-`*` + client-only.** No backend, no API keys at build time. A source that needs a
  key or blocks CORS is out (or proxied later, which is out of scope now).
- **Best-effort must never break the home screen.** The existing "never throw into Home" contract
  stays. The change is that failure becomes *visible status*, not *invisible blank*.
- **Auto-deploy.** Every push to `main` ships to prod (GitHub Pages). The whole current livefire v2
  is uncommitted; increments must keep `npm run verify` (incl. `verify:livefire`) and `npm run build`
  green before any push.

## Premises

Confirmed this session (owner agreed with the framing and the recommended path):

- **P1 — Kill silent degradation first.** Per-layer status is **live (published Xh ago) / cached (Xh
  old) / unavailable**. This single change fixes "active shows nothing," "missing fires," and
  "all-clear" looking identical. Non-negotiable for "trust is utmost"; it is step one.
- **P2 — "Last updated" = the SOURCE's publish time, not our fetch time.** Replace `relTime(fetchedAt)`
  with the real source timestamp per layer (CWFIS scan/`rep_date`, CIFFC `field_situation_report_date`/
  `field_status_date`, FWI issue time, M3 `lastdate`, SPSA edit time, SaskAlert sent time).
- **P3 — More sources ≠ more trust.** Add provincial data only where reliable + high-signal. SK
  (SPSA) earns its place first because the game lives in northern SK; no Canada-wide scraper sprawl.
- **P4 — No-scroll law still rules.** Adding smoke + scrubber + four layers means the layer control
  must be redesigned compact, not extended.
- **P5 — Every total names its window.** Label "today" vs "this year" vs "all-time" on the number
  itself; add **this year vs the 10-year average** if the source exposes it (the one context every
  real wildfire dashboard shows).
- **P6 — "Active fire shows nothing" is likely a real broken endpoint.** The authoritative layer hangs
  on one CIFFC WFS call already flagged fragile in project memory. Verify it live and harden or swap
  (SK SPSA's ArcGIS active-fire feed is a strong, CORS-friendly backstop) — step zero, before adding.

## Approaches Considered

### Approach A: Trust spine first, then layers one at a time (CHOSEN)
Build the source-status + honest-freshness spine and fix/harden the existing four (including
un-breaking active fires); ship that. Then add SPSA → smoke → bans → evac incrementally on a uniform
`LiveSource` abstraction, each landing only when honest. The source-ledger folds into the spine as the
trust hero. Effort: M (spine) then S/layer. Risk: Low. Reuses all of `normalize.ts`'s defensive
parsers, `fetchJsonCached`, `FeedSource`, the lazy-Leaflet pattern, and `verify:livefire`.

### Approach B: Big-bang full window
Build the abstraction + all four new layers + the smoke scrubber + the spine in one push; ship when
whole. Effort: XL. Risk: High — a long red branch where the two fragile feeds (bans/evac) can block the
entire ship; hard to verify; fights the repo's "ship small, stay green" grain.

### Approach C: Honesty-as-hero + time replay
Lead with the source-ledger + a fire/smoke time-scrubber "replay" ("watch the fire breathe over the
last 48h") as the centerpiece. Effort: L. Risk: Med — highest delight and most on-brand, but the
time-dimension is the most net-new engineering and tempts polishing before the data is honest.

## Recommended Approach

**A, with C's source-ledger folded into the spine** (owner-approved). Because trust is utmost, fragile
evac/bans data must not ship until the spine exists to caveat it; incremental keeps Home and the CI
gates green at every step. The cheap, high-payoff piece of C (the source-ledger) is part of the spine.
The time-replay from C is a deliberate *later* add, only once the layers are honest.

### The spine (the `LiveSource` model)

Generalize today's bespoke per-feed wiring into one uniform unit so new layers plug in without churn:

```
// fetch() resolves to ONE snapshot — there is no second status field.
LiveSource<T>.fetch(force) -> {
  id, label, data: T,
  status: 'live' | 'unavailable' | 'disabled',  // fetch-level outcome (see below)
  fromCache: boolean,        // orthogonal: did this snapshot come from the localStorage cache?
  publishedAt: epoch ms | 0, // from the SOURCE (drives ALL user-facing freshness copy)
  fetchedAt:   epoch ms,     // when WE called (cache-TTL math only — never shown)
  attribution, sourceUrl,
  disclaimer?: string,       // per-layer caveat (forecast / not-for-emergency / link-out)
}
```

- **`status` is fetch-level only**: `live` (got data, fresh or from cache), `unavailable` (failed/
  timed out, no usable data), `disabled` (the `VITE_LIVEFIRE_DISABLE` kill-switch — a distinct state,
  NOT "down", so the ledger never claims an intentionally-off feed is broken).
- **`fromCache` is orthogonal to `status`.** A feed can be `status:'live'` + `fromCache:true` (we
  served a cache hit within TTL). The user-facing string is always derived from `publishedAt`
  ("published 3h ago"), with at most a quiet "from cache" hint — we never show *our* cache age as if
  it were the data's age.
- **`empty` is NOT a fetch-level status — it's a UI-layer computation.** "No fires in view" depends on
  the country filter (`menus.ts` filters AFTER fetch) and the map viewport, neither of which the
  source knows. So the UI computes, per layer, per current filter: feature-count === 0 → render
  **"none in view"**; `status==='unavailable'` → render **"feed unavailable"**; `status==='disabled'`
  → hide the chip. The core trust fix is this three-way distinction at the UI layer — *empty ≠ down ≠
  off* — not a fourth enum on the source.
- `publishedAt` drives all freshness copy. `fetchedAt` stays only for cache-TTL math.
- **WMS layers (FWI) have no per-feature JSON timestamp.** Their `publishedAt` must come from a WMS
  `GetCapabilities` time-dimension parse (or the dated layer name). If neither is cheaply available,
  show an honest "current conditions (issue time unavailable)" rather than faking a timestamp — this
  is materially more work than reading a JSON field, so budget for it in Increment 1.
- The **Source ledger** (a small panel, openable from a status chip in the firebar) lists every
  `LiveSource`: label · status dot · "published Xh ago" · link to origin · disclaimer. This is the
  trust hero and where the "not for emergency decisions — see official sources" line lives.
- Each map layer's toggle chip shows a status dot **derived from the same per-layer state** the ledger
  uses (one source of truth): live / from-cache / unavailable / none-in-view. A dead layer reads as
  *unavailable*, an off toggle reads as *off*, an empty-in-view layer reads as *none in view* — three
  visibly different things.

### Candidate data sources (verify endpoint + CORS live during build — do NOT trust this list blind)

- **Existing, keep + make honest:** CWFIS `hotspots_last24hrs`, CWFIS `m3_polygons_current`,
  CWFIS `fwi_current` (WMS), CIFFC `/v1/dashboard/summary`, CIFFC `ytd_fires` (WFS).
- **Active-fire fix (P6):** verify `ciffc:ytd_fires` live; add **SK SPSA active-wildfire ArcGIS
  FeatureServer** as the SK-authoritative source and as a backstop if CIFFC WFS is the broken link.
- **Smoke (forecast, animated):** firesmoke.ca / BlueSky-FireWork imagery, or NOAA HRRR-Smoke. Confirm
  a usable tile/WMS/image-sequence endpoint + CORS; label clearly as a **forecast**, with frame
  timestamps for the scrubber.
- **Fire bans:** SK provincial fire/open-fire restriction status. Scope to **provincial-level** status,
  not every municipality. Likely the most fragile — keep it opt-in and loudly honest when stale.
- **Evac alerts/orders:** **SaskAlert** (the provincial public alerting system; CAP/structured feed is
  far better than scraping news). Distinguish **ALERT vs ORDER**, attribute + link to the official
  source, never present as authoritative — the screen *surfaces* the alert and *points* to the origin.
  **CORS is the open risk here** (provincial alert feeds are often built for server-side polling and
  may not send `Access-Control-Allow-Origin: *`). If SaskAlert is CORS-blocked, the no-proxy constraint
  means we do **not** fetch it live — fall back to a **link-out-only** evac entry in the ledger (a
  button to the official SaskAlert page, no live data, clearly labeled). That still honors "surfaces +
  points to origin" without faking freshness, and it means a CORS block degrades the increment rather
  than deleting it.

### Freshness + "this year / all years" copy

- Per-layer: "Hotspots — satellite pass 3h ago," "Active fires — agency report 1h ago," etc.
- Stat strip: label each number's window inline ("Active now," "This season," "Burned this season").
  Add **"this season vs 10-yr avg"** if a source exposes it (CIFFC situation report) — else omit
  rather than fabricate. If the owner wants *all-time*, note no keyless source guarantees it; flag as
  open question rather than invent.

### FWI opacity

Ship the cheap fix first: **lower the default** (e.g. ~0.22–0.28) so the basemap stays readable — that
alone satisfies the success criterion. Add an **opacity slider** on the FWI chip only if dogfooding
still complains (it's the one piece that exceeds the stated problem, and the FWI layer is opt-in /
default-off, so the slider is a nice-to-have, not a requirement). Either way the value lives where the
other livefire tuning would — keep it out of module hard-coding per the repo's config convention (there
is no livefire block in `config.ts` today; add one).

## Open Questions

- Does CIFFC `ytd_fires` WFS actually work right now, or is it the broken endpoint behind "active shows
  nothing"? (Verify live first — this decides whether SPSA is a fix or just an addition.)
- Is there a keyless, CORS-`*` smoke endpoint that supports an animated frame sequence, or only a single
  current frame? (Decides whether the scrubber is in the smoke increment or deferred.)
- Does any keyless source expose the **10-year-average** comparison? If not, label this-season clearly
  and drop the comparison rather than fake it.
- "All years / all-time" totals: no obvious keyless source — confirm whether the owner wants this badly
  enough to justify a one-off data file, or whether "this season + 10-yr avg" satisfies the intent.
- **SaskAlert keyless/CORS availability** (Increment 6): if it does not send CORS `*`, evac becomes
  link-out-only (no live fetch) under the no-proxy constraint — confirm that fallback is acceptable
  before building Increment 6.
- **WMS publish-time** (FWI, and any future WMS layer): is a `GetCapabilities` time-dimension parse
  worth it, or is "current conditions (issue time unavailable)" an acceptable honest label?
- Fire bans data shape in SK — structured feed or HTML scrape? (Scrape = fragile = strongest candidate
  to defer or keep behind an honest "may be out of date" badge.)

## Success Criteria

- "Active fire shows nothing" is gone: when the active-fire feed is down, the layer chip + ledger say
  **unavailable**; when there are genuinely no active fires in view, they say **none in view** — never
  a silent blank.
- Every layer shows **source publish time**, not fetch time; a stale feed says so.
- FWI raster no longer washes out the map (lower default + slider), basemap legible underneath.
- Every stat names its time window; no number is ambiguous about "today vs this year." The strip shows
  **today + this-season (+ this-season-vs-10-yr-avg if a source exposes it)**. **All-time is explicitly
  declined for now** — no keyless source guarantees it, so rather than fabricate a number we either omit
  it or tell the user it isn't available (this is a deliberate narrowing of the owner's "all years" ask,
  revisited only if a real source is found or a one-off data file is justified).
- SK fires, smoke, bans, and evac are each present, each honestly statused, each link out to origin,
  each carrying the right caveat (forecast / not-for-emergency).
- The screen fits a phone with **zero page scroll**.
- `npm run verify` (incl. `verify:livefire`) and `npm run build` stay green at every increment; each
  increment is independently shippable.
